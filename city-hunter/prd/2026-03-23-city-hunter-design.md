## 一、项目概述

### 1.1 产品定位

城市猎人是一款基于地理位置的城市解谜小程序。玩家通过对话形式推进剧情，在对话中获取线索，线索指向真实的城市地点。玩家前往实地探索地标和文化景点，通过现场寻找线索解开谜题，最终完成整个剧情。

### 1.2 核心特色

- **实地探索**：结合真实城市地标，增强沉浸感
- **剧情驱动**：分支对话 + 多重结局，重玩价值高
- **实体道具**：部分案件配套实物道具，增强代入感
- **线索推理**：地点不直接展示，玩家需从对话中推理

### 1.3 目标平台

- **一期**：微信小程序（验证核心玩法）
- **二期**：独立 App（扩展功能）

### 1.4 商业模式

案件付费解锁，部分免费案件吸引用户，高级案件需付费。

### 1.5 首发策略

先在上海深耕，打磨产品体验，后期扩张至其他城市。

## 二、整体架构

### 2.1 系统分层

```
┌─────────────────────────────────────────────────────┐
│                    客户端层                          │
│   Uni-app 小程序/App (Vue3 + TypeScript)            │
├─────────────────────────────────────────────────────┤
│                    云服务层                          │
│   uniCloud 云函数 + 云数据库 + 云存储                │
├─────────────────────────────────────────────────────┤
│                    业务服务层                        │
│   用户服务 │ 案件服务 │ 对话引擎 │ 游戏服务 │ 订单服务 │
└─────────────────────────────────────────────────────┘
```

### 2.2 技术选型

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 小程序端 | Uni-app (Vue3) | 跨端支持 |
| 用户系统 | uni-id | 官方用户系统 |
| 支付系统 | uni-pay | 微信支付封装 |
| 云函数 | uniCloud (阿里云) | 无服务器架构 |
| 数据库 | uniCloud 云数据库 | MongoDB 兼容 |
| 存储 | uniCloud 云存储 | OSS 封装 |
| 后台管理 | Vue3 + Element Plus | Web 管理后台 |

### 2.3 核心模块划分

| 模块 | 职责 | 核心数据表 |
|------|------|------------|
| 用户模块 | 注册登录、个人档案、游戏进度 | users, user_progress |
| 城市模块 | 城市列表、城市详情 | cities |
| 案件模块 | 案件列表、案件详情、购买记录 | cases, orders |
| 对话引擎 | 对话树解析、分支选择、多媒体渲染 | dialogue_nodes, dialogue_choices |
| 线索模块 | 线索收藏、笔记管理 | user_notes |
| 地理模块 | GPS定位、地点验证、打卡记录 | locations |
| 支付模块 | 案件购买、订单管理 | orders |

## 三、核心数据模型

### 3.1 城市与案件

```
cities（城市）
├── _id: 城市ID
├── name: 城市名称
├── cover: 封面图URL
├── description: 城市介绍
├── status: 状态（active/inactive/coming_soon）
├── coordinates: { lat, lng }
└── sort_order: 排序权重

cases（案件）
├── _id: 案件ID
├── city_id: 所属城市
├── title: 案件名称
├── cover: 封面图URL
├── description: 案件简介
├── difficulty: 难度等级（1-5星）
├── duration_minutes: 预估时长
├── price: 价格（分）
├── location_count: 地点数量
├── start_node_id: 对话起始节点
├── has_props: 是否包含实体道具
├── status: 状态（draft/active/inactive）
└── sort_order: 排序权重
```

### 3.2 对话系统

```
dialogue_nodes（对话节点）
├── _id: 节点ID
├── case_id: 所属案件
├── node_type: 节点类型（text/image/video/choice/location_check/answer_verify/ending）
├── speaker: 说话人（NPC名称）
├── speaker_avatar: 说话人头像
├── content: 内容（JSON）
├── auto_next: 自动推进目标节点
├── next_node_id: 下一节点（验证节点）
├── location_id: 关联地点ID
├── correct_answers: 正确答案数组
├── error_hint: 错误提示
├── hint_content: 该节点提示内容
├── ending_id: 关联结局ID
├── position_x/y: 编辑器坐标
└── created_at: 创建时间

dialogue_choices（对话选项）
├── _id: 选项ID
├── node_id: 所属节点
├── choice_text: 选项文案
├── next_node_id: 跳转目标节点
├── required_conditions: 前置条件（JSON）
├── tag: 分支标记（用于结局计算）
└── sort_order: 排序
```

### 3.3 地点与结局

```
locations（地点）
├── _id: 地点ID
├── case_id: 所属案件
├── name: 地点名称
├── address: 详细地址
├── latitude/longitude: GPS坐标
├── radius_meters: 打卡半径（默认50米）
├── hint: 线索提示
├── cover: 地点封面
├── description: 文化介绍
└── order_index: 探索顺序

endings（结局定义）
├── _id: 结局ID
├── case_id: 所属案件
├── title: 结局标题
├── description: 结局描述
├── image: 结局图片
├── rarity: 稀有度（common/rare/hidden）
└── conditions: 触发条件（JSON）
```

### 3.4 用户进度

```
user_progress（用户案件进度）
├── _id: 记录ID
├── user_id: 用户ID
├── case_id: 案件ID
├── current_node_id: 当前对话节点
├── visited_nodes: 已访问节点数组
├── check_in_locations: 已打卡地点数组
├── branch_state: 分支状态记录（JSON）
├── hints_used: 已使用提示次数
├── is_completed: 是否完成
├── ending_id: 达成的结局
├── play_time_seconds: 游戏时长
└── last_play_at: 最后游玩时间

user_notes（用户笔记）
├── _id: 笔记ID
├── user_id: 用户ID
├── case_id: 案件ID
├── content: 收藏的对话内容
├── source_node_id: 来源节点
├── annotation: 用户标注
└── created_at: 创建时间
```

### 3.5 订单与地址

orders（订单）
├── _id: 订单ID
├── order_no: 订单编号
├── user_id: 用户ID

## 四、核心业务流程

### 4.1 案件购买流程

```
用户选择案件 → 检查购买状态
    ↓
未购买：检查是否有实体道具
    ├── 有：引导填写收货地址 → 发起支付
    └── 无：直接发起支付
    ↓
支付成功 → 创建进度记录（current_node_id = start_node_id）
    ↓
允许开始游戏
```

### 4.2 对话推进流程

```
展示当前节点内容
    ↓
玩家可随时长按对话 → 收藏到笔记本
    ↓
判断节点类型
    ├── 普通对话：自动推进到下一节点
    ├── 选择分支：显示选项，记录选择，跳转目标节点
    ├── 地点打卡：进入打卡流程
    ├── 答案验证：进入验证流程
    └── 结局节点：计算结局，结束游戏
```

### 4.3 分支选择流程

```
当前节点有分支选项
    ↓
检查每个选项的前置条件
    ├── 条件满足：显示该选项
    └── 条件不满足：隐藏该选项
    ↓
玩家从可见选项中选择
    ↓
记录分支选择到 branch_state
    ↓
跳转到 next_node_id
```

### 4.4 地点打卡流程

```
对话推进到 location_check 节点
    ↓
展示线索提示（无地点名称）
    ↓
玩家线下探索，推理目标地点
    ↓
玩家到达后点击"验证地点"
    ↓
获取当前GPS坐标
    ↓
遍历该案件所有 locations，计算距离
    ↓
    ├── 找到匹配地点（距离≤50米）：打卡成功，展示地点详情
    └── 无匹配地点：提示"这里似乎不是目标地点"
    ↓
更新进度，解锁后续对话
```

### 4.5 答案验证流程

```
对话推进到 answer_verify 节点
    ↓
展示问题和答案来源提示
    ↓
玩家输入答案
    ├── 正确：验证成功，继续剧情
    └── 错误：错误计数+1
        ├── 次数<3：提示重试
        └── 次数≥3：提供额外提示
```

### 4.6 提示系统流程

```
对话界面显示"提示"按钮（显示剩余次数）
    ↓
玩家点击提示 → 弹窗确认
    ↓
确认后扣减提示次数
    ↓
根据当前节点类型返回对应提示
    ├── 普通对话：提示后续剧情方向
    ├── 地点打卡：提示地点方向/区域
    └── 答案验证：提示答案相关线索
```

### 4.7 结局计算流程

```
对话推进到结局节点
    ↓
读取 user_progress.branch_state
    ↓
查询该案件所有 endings
    ↓
按 conditions 匹配
    ├── 匹配成功：达成特定结局
    └── 无匹配：达成默认（普通）结局
    ↓
更新进度（is_completed=true, ending_id）
    ↓
展示结局画面 + 游戏统计 + 成就
```

### 4.8 线索收藏流程

```
对话界面展示
    ↓
玩家长按对话消息
    ↓
进入多选模式（类似微信）
    ↓
勾选一条或多条对话
    ↓
点击"加入笔记本"
    ↓
可添加个人标注
    ↓
保存到 user_notes
```

## 五、对话脚本格式

### 5.1 基础结构

```
# 案件：外滩迷踪
城市：上海
版本：1.0.0

---

## node_001

[type]: text
[speaker]: 侦探张三
[content]:
昨天晚上你在哪里？

我注意到他的眼神闪烁了一下。

[auto_next]: node_002

---

## node_002

[type]: choice
[content]:
你打算怎么做？

[options]:
- "告诉警察真相" -> node_005
  tag: report_truth

- "帮嫌疑人隐瞒" -> node_008
  tag: hide_truth

---

## node_003

[type]: location_check
[location_id]: loc_001
[hint]:
那座古老的石桥下，流水声中似乎藏着什么秘密...

[single_hint]: 搜索"外婆桥"，坐标在黄浦区...

---

## node_004

[type]: answer_verify
[question]:
信件第三段提到的日期是多少？

[source_hint]:
请查看您收到的道具信件

[answers]:
- 1937.08.13
- 1937年8月13日

[single_hint]: 答案在信件右上角的邮戳上

---

## node_099

[type]: ending
[ending_id]: ending_001
```

### 5.2 节点类型属性

| 节点类型 | 必填属性 | 可选属性 |
|----------|----------|----------|
| text | type, speaker, content | auto_next, hint_content |
| image | type, url | description, auto_next |
| video | type, url | description, auto_next |
| choice | type, options | content |
| location_check | type, location_id, hint | hint_content |
| answer_verify | type, question, answers | source_hint, error_hint, hint_content |
| ending | type, ending_id | - |

### 5.3 条件语法

```
[options]:
- "选项文案" -> target_node_id
  condition: node_XXX.completed, check_in.location_XXX
  tag: branch_tag
```

## 十一、运营指南

### 11.1 案件创建流程

```
1. 策划阶段
   ├── 确定案件主题和背景故事
   ├── 设计剧情大纲和分支结构
   ├── 规划地点线索和探索路径
   └── 撰写完整对话脚本

2. 实地踩点
   ├── 确认地点可进入性
   ├── 精确测量GPS坐标
   ├── 拍摄地点照片
   └── 收集验证问题和答案

3. 内容录入
   ├── 创建城市和案件
   ├── 录入地点信息
   ├── 创建对话节点
   └── 配置结局条件

4. 测试验收
   ├── 内部测试完整流程
   ├── 实地测试打卡功能
   └── 验证所有分支路径

5. 上线发布
   ├── 设置案件状态为上架
   └── 监控上线后数据
```

### 11.2 日常运营指标

| 指标 | 目标值 |
|------|--------|
| 付费转化率 | > 5% |
| 案件完成率 | > 60% |
| 次日留存 | > 40% |

| 平均评分 | > 4.5 |

### 11.3 应急处理
